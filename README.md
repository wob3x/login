# Архитектура безопасной передачи данных: паттерн Backend-for-Frontend (BFF) с использованием Serverless

## Цель
Спроектировать отказоустойчивую архитектуру для сбора данных с клиентской части и их маршрутизации в защищенный контур (внешний API) с полным сокрытием служебных ключей аутентификации от конечного пользователя.

## Компоненты системы

1. **Точка входа (Frontend / Presentation Layer)**  
   Размещается на статическом хостинге (например, GitHub Pages). Отвечает исключительно за отрисовку интерфейса, сбор данных и их отправку на промежуточный шлюз.

2. **Промежуточный шлюз (Middleware / API Gateway)**  
   Размещается в бессерверной среде (например, Cloudflare Workers). Выступает в роли прокси-сервера. Его задача — принять запрос от фронтенда, валидировать его, обогатить скрытыми учетными данными и перенаправить в конечную точку.

3. **Конечная точка (Backend / Data Receiver)**  
   Внешняя инфраструктура (Telegram API), принимающая итоговую полезную нагрузку.

## Что должно быть в проекте

- `index.html` — клиентская часть (UI + отправка запроса в BFF).
- `_next/` — статические ассеты интерфейса.
- `worker.js` — BFF-шлюз (валидация, CORS, санитизация, проксирование в Telegram API).
- `.env` — локальные переменные для разработки.
- `.dev.vars.example` — пример переменных для локального запуска Worker.
- `.gitignore` — исключение секретов из репозитория.

## Пошаговый план развертывания

### Этап 1: Подготовка конечной точки (Data Receiver)

1. Инициализация принимающего инстанса (регистрация бота через платформу провайдера).
2. Генерация токена доступа и получение уникального идентификатора канала/чата.
3. Классификация токена и `chat_id` как секретов.

### Этап 2: Изоляция секретов и настройка шлюза (Middleware)

1. Создание бессерверного приложения (`worker`) в облачном контуре.
2. **Secrets Management**: сохранение токена и идентификатора чата в зашифрованные переменные окружения платформы.  
   Это исключает появление hardcoded secrets в исходных файлах.
3. Настройка маршрутизации и валидации:
   - прием только `POST`-запросов;
   - обработка `OPTIONS` для CORS preflight;
   - парсинг и валидация входящего JSON;
   - экранирование входящих данных для защиты от инъекций спецсимволов.
4. Настройка отправки сформированного пакета данных с сервера шлюза на сервер конечной точки.

### Этап 3: Проектирование клиентской части (Frontend)

1. Разработка статического веб-интерфейса.
2. Настройка асинхронной передачи: при выполнении целевого действия интерфейс формирует пакет данных и отправляет его строго на URL промежуточного шлюза.
3. Полное отсутствие прямых упоминаний конечной точки (Telegram API) на клиенте.
4. Интеграция TWA SDK: подключение библиотек провайдера для адаптации интерфейса под нативное окружение мессенджера (цветовые схемы, внутренние методы рендеринга).

### Этап 4: Настройка контроля доступа и защита (Hardening)

1. Ограничение CORS: конфигурирование `Access-Control-Allow-Origin` так, чтобы шлюз принимал запросы только с доверенного домена (развернутого на Этапе 3).
2. Проверка заголовков: внедрение проверки `Referer`/`Origin` на стороне шлюза для блокировки запросов от сторонних источников или скриптов автоматизации.
3. Рекомендуемо дополнительно:
   - rate limiting;
   - логирование подозрительных запросов;
   - аудит ошибок валидации.

## Резюме для защиты архитектуры

Предложенная топология решает критическую уязвимость клиентских веб-приложений — компрометацию API-ключей. Благодаря вынесению бизнес-логики в Serverless-контур:

1. **Инфраструктура скрыта**  
   При анализе сетевого трафика (через DevTools или снифферы) аналитик видит только обращение к абстрактному облачному шлюзу.

2. **Секреты защищены**  
   Ключи авторизации физически не передаются в браузер пользователя, что делает бессмысленным статический анализ клиентского кода.

3. **Вектор атак сужен**  
   Настроенные политики CORS и проверка заголовков исключают возможность несанкционированного использования шлюза сторонними лицами.

## Важное замечание

Проверка `Origin`/`Referer` повышает защищенность, но не является абсолютной защитой в небраузерных сценариях.  
Для production-релиза рекомендуется дополнять архитектуру signed requests (HMAC/JWT), rate limiting и мониторингом.

## Итоговый чек-лист (0$)

1. **Бот создан** в `@BotFather` (бесплатно).
2. **Worker развернут** на Cloudflare (бесплатно).
3. **Токен и ChatID спрятаны** в Environment Variables воркера.
4. **Сайт залит** на GitHub Pages (бесплатно).
5. **Домен GitHub** привязан к Cloudflare (опционально, для красоты).

## Практическая настройка для текущей реализации

### 1) Переменные Worker

Настройте в Cloudflare Worker:

- `BOT_TOKEN` (Secret)
- `ADMIN_JOIN_PASSWORD` (Secret)
- `ADMIN_CHAT_ID` (Variable, можно CSV: `123,456`)
- `ALLOWED_ORIGINS` (Variable, можно CSV доменов)

Опционально для динамических админов:

- KV binding `SUPPORT_KV`  
  (хранит админов, которые ввели пароль в боте через `/admin`).

### 2) Telegram webhook

После деплоя Worker выполните запрос:

`POST https://<your-worker-domain>/telegram/set-webhook`

Это привяжет webhook к `https://<your-worker-domain>/telegram/webhook`.

### 3) Как админу подключиться через бота

1. Открыть бота и нажать `/start`.
2. Отправить: `/admin YOUR_PASSWORD`.
3. После успешной проверки чат станет админским:
   - c KV: сохраняется в `SUPPORT_KV`
   - без KV: добавляйте `chat_id` вручную в `ADMIN_CHAT_ID`.

### 4) Фронтенд

В `index.html` укажите:

`const API_BASE = "https://<your-worker-domain>";`

Фронтенд отправляет:

- `POST /api/visit` при открытии страницы
- `POST /api/support` при отправке формы (`username + question`)
